# 心镜 XinJing — 四类界面问题定位文档

> 日期：2026-07-15　版本：3.7.1　状态：已定位根因，待修复

本文把用户反馈的 4 类问题逐条写清「现象 / 根因（文件·行号）/ 修复方向」，作为后续修复的依据。

---

## 问题 1：任何"选会话历史"的界面都会出现错误节次

### 现象
在需要选来访者会话的界面（如 AI 督导 → 选来访者「白」），会话列表里同时出现：
- `第1节 6-30`（错误，不该出现）
- `第25节 6-30`（正确）

即同一天出现了两条不同节次的会话，其中一条是错误/重复记录。

### 根因
1. **列表去重维度不足**：各选会话列表只按 `session.id` 去重，没有按「日期 + 节次」去重，因此两条不同 id、但落在同一天的会话会同时显示。
   - `app/js/supervision.js:113-115`：`getSessionsByClient(cid)` 后仅 `seenSup[id]` 去重。
   - `app/js/supervision.js:143`：直接输出 `第${s.sessionNumber}节`，重复条目都会渲染。
   - `app/js/consult-notes.js:329-339`、`report-writing.js`、`doc-center.js`：同源写法。
2. **数据源存在重复/孤儿会话**：
   - `store.js:427-432` `nextSessionNumber` 返回 `max(existing)+1`，正常应连续；出现「第1节」说明历史上曾产生过一条 sessionNumber=1 的脏记录（早期批量记账、seed、sync，或 client 删除未级联时残留）。
   - `store.js:388-405` `deleteClient` 现已级联清理 `cache.sessions`，故新数据不再孤儿；但**旧脏数据仍留在库里**。

### 修复方向
- **表现层**：各选会话列表在按 id 去重基础上，追加按 `date + sessionNumber` 去重（同日同号只留一条；同日不同号保留），并可对明显是账单存根（`billing.source==='manual'` 且 durationMinutes=0 且无逐字稿）做过滤或标注。
- **数据层**：提供一次性「会话体检」清理——扫描每个来访者，合并/删除同日重复、修正断裂节次编号。

## 问题 2：账单"记一笔"节数 > 1 时日期不自动递增

### 现象
账单 →「记一笔」弹窗里「节数」可手动改为 >1，期望是保存后每一节的日期**依次加一天**；实际多条记录全部堆在同一天。

### 根因
`billing-shell.html:930-939` 批量建 session 时，日期变量在循环内固定不变：
```js
dates.forEach(function (date) {
  var nextNo = Store.nextSessionNumber(cid);
  for (var i = 0; i < snum; i++) {
    Store.createSession({
      clientId: cid, date: date, sessionNumber: nextNo + i, ...  // ← date 恒为同一天
    });
  }
});
```
另一处批量入口 `binAddSession`（约 `billing-shell.html:1879-1883`）同样 `date: date` 固定。

### 修复方向
循环体内把日期改为 `addDays(date, i)`（第 i 节 = 起始日期 + i 天），两处批量入口同步修。需新增一个本地 `addDays(dateStr, n)` 工具（按 YYYY-MM-DD 解析后加天数再格式化，避免时区偏移）。

## 问题 3：已激活会员，账单/咨询日历等界面仍显示"激活会员"

### 现象
用户已激活会员，但账单界面、咨询日历等页面顶部仍出现「试用中 · 激活会员」配额条。

### 根因
顶栏配额条注入存在**竞态 + 未订阅状态变化**：
- `app/js/app.js:952-983` `injectQuotaBar()` 的 `build()` 在 `DOMContentLoaded + 100ms` 执行；
- 判断是否隐藏依赖 `App.aiUnlocked()`（`app.js:970`），而 `aiUnlocked()` 读的是 `licenseStateCache`；
- `licenseStateCache` 初值来自 `window.__XJ__`（`app.js:105`），这是 **preload 初始化快照，激活后不会自动同步**；
- 权威状态要靠 `refreshLicenseState()`（`app.js:114-121`，`await __XJ_API__.getState()`）拉取，但它只在 `initPage` 里调用——**build() 常常早于 refresh 执行**，此时读到的是 stale 快照（未激活），于是照样显示"激活会员"。
- 且 build() **没有注册 `onLicenseStateChange`**（`app.js:163`），激活状态后续广播过来也不会重绘/移除配额条。
- 附带逻辑缺陷：`app.js:970-972` 仅当 `tier ∈ {pro,full,custom}` 才隐藏，试用期已解锁（trial + aiUnlocked）用户不会被隐藏。

### 修复方向
- `build()` 改为**先 `await refreshLicenseState()` 再判定**，确保读到权威授权状态；
- 注册 `onLicenseStateChange`，状态变化时移除/重建配额条；
- 隐藏条件放宽为「`aiUnlocked()` 为真即隐藏」（含试用期解锁），或明确覆盖 pro/full/custom/trial-unlocked。

## 问题 4：日历——大量"已删除" + 不支持重复预约 + 删除粒度单一

### 现象
1. 咨询日历里出现大量「（已删除）」条目（孤儿会话）。
2. 新建对话不支持按周 / 双周 / 按月重复预约。
3. 删除只有单条删除，缺少「删除本次 / 删除全部 / 删除本次及之后」三种粒度。

### 根因
- **孤儿残留**：`session-calendar.js:121` `clientName` 回退 `'（已删除）'`——说明库里存在 client 已删但 session 未随删的历史数据（早期 `deleteClient` 未级联时期产生）。
- **无重复机制**：`session-calendar.js:468-604` `openNewSession` 表单只有 日期/时间/类型/状态，无重复选项；save（`:556-603`）直接 `Store.createSession` 单条，无 `seriesId`/`recurrence` 概念。
- **删除粒度单一**：`session-calendar.js:457-464` `removeSession` 仅 `confirm('确认删除这节会话？')` + `Store.deleteSession(sessionId)`，无系列概念、无三选项。

### 修复方向
- **重复预约**：`openNewSession` 增加「重复」选项（不重复 / 每周 / 每两周 / 每月 + 结束日期或次数）；save 时展开为多条会话，每条带同一 `seriesId` 与 `recurrence` 规则。
- **删除粒度**：`removeSession` 改为弹窗三选项——删除本次（仅该条）/ 删除本次及之后（同 seriesId 且 date ≥ 当前）/ 删除全部（整个 seriesId）。无 seriesId 的旧数据退化为单条删除。
- **孤儿清理**：提供一次性扫描，删除 clientId 已不存在的 session（与问题 1 的数据体检合并实现）。

---

## 修复顺序建议
1. 问题 2（改动最小、纯 bug）→ 2. 问题 3（授权竞态）→ 3. 问题 1（去重 + 数据体检）→ 4. 问题 4（重复预约 + 删除粒度，改动最大）。
4 项都涉及 Store/日历数据，建议问题 1 与问题 4 的「孤儿/重复清理」合并为同一个数据体检函数。
