# 心镜 XinJing 安装报错 "Failed to uninstall old application files.:2" 排查与修复

> 适用版本：3.6.7 / 3.6.8 自动更新升级失败
> 撰写：2026-07-16（基于 electron-builder NSIS 模板源码定位）
> 结论：本错误与"无法关闭心镜"是**两个不同环节**，属升级模式的"卸载旧版目录"步骤失败。

---

## 一、错误含义

```
Failed to uninstall old application files. Please try running the installer again.:2
```

- 这条信息来自 electron-builder 的 `handleUninstallResult`（`app-builder-lib/templates/nsis/include/installUtil.nsh:128-132`）。
- 结尾的 `:2` 是**旧版卸载程序（old-uninstaller.exe）返回的退出码 `$R0`**。源码逻辑：

  ```nsis
  ${if} $R0 != 0
    MessageBox MB_OK|MB_ICONEXCLAMATION "$(uninstallFailed): $R0"   ; 即本错误信息 + ":2"
    SetErrorLevel 2
    Quit
  ${endif}
  ```

- 也就是说：**新安装器在升级模式下，先去调用旧版本的卸载器来删旧目录，而旧卸载器自己以退出码 2 失败了，于是新安装器中止并报此错。**

---

## 二、为什么会失败（根因）

升级流程（`uninstallOldVersion`，`installUtil.nsh:142-243`）：

1. 从注册表读取旧卸载器路径 `UninstallString` 与安装位置 `InstallLocation`；
2. 把旧卸载器复制到临时目录，以静默方式运行：`old-uninstaller.exe /S /KEEP_APP_DATA ... _?=$installationDir`；
3. 旧卸载器（`uninstaller.nsh`）先尝试把整个 `$INSTDIR` **原子重命名**为 `old-install`，若其中任一文件被占用 → `Abort`；随后 `RMDir /r $INSTDIR`；
4. 旧卸载器 abort → 退出码 2 → 新安装器弹错并 `Quit`。

**关键事实**：用户已用脚本确认机器上**没有任何 `xinjing.exe` / 安装器进程**（`tasklist` 返回"没有运行的任务匹配指定标准"）。因此占用**不是心镜自身进程**，而是**外部进程持有旧安装目录里的文件句柄**，常见三类：

| 嫌疑 | 说明 |
|---|---|
| A. Windows Defender 实时防护 | 删除瞬间防病毒会短暂锁文件，导致原子重命名/删除失败 |
| B. 资源管理器 / 外壳扩展 | 若 `AppData\Local\xinjing-updater` 在 Explorer 中打开，或缩略图缓存持柄 |
| C. 备份 / 同步 / 索引代理 | 第三方备份、云端同步、Windows Search 索引服务持柄 |

> 注：3.6.8 新增的 `build/installer.nsh`（`customCheckAppRunning`，负责"进程占用时中文指引 + 重试强杀"）**只作用于安装前的"应用是否在运行"检查**，与本错误无关。你已越过那一步，说明 3.6.8 的修复生效；现在卡在更后面的卸载步骤。

---

## 三、可靠修复（让安装器跳过旧卸载器）

原理：只要注册表里不再有 xinjing 的卸载项，安装器读不到 `UninstallString`，`uninstallOldVersion` 会提前 `Return`（`installUtil.nsh:156-163`），走**全新安装**分支，不再依赖会 abort 的旧卸载器。

> 以下操作均在**你自己那台装了桌面客户端的机器**上执行。删除的是应用安装目录与注册表项，**不涉及任何个人文档/数据**（用户资料库在 `AppData\Roaming\xinjing`，本操作不删它）。

### 步骤 1：确认无残留进程
任务管理器确认没有 `xinjing.exe` / `xinjing-setup` 进程（你已确认过）。如有，先结束。

### 步骤 2：删除旧安装目录
旧安装目录为（按你的排查脚本实测）：

```
C:\Users\Administrator\AppData\Local\xinjing-updater\
```

- 先把里面的 `pending\xinjing-setup-3.6.8.exe` **复制出来**（例如放到桌面），后面要用；
- 然后删除整个 `xinjing-updater` 目录。
- 亦可顺手清理 `AppData\Local\Programs\xinjing-updater`（若不存在可忽略）。

### 步骤 3：删除卸载注册表项（关键）
`Win + R` → `regedit`，删除以下项（per-user 安装，重点在 HKCU；若曾以管理员装过，也查 HKLM）：

```
HKEY_CURRENT_USER\Software\Microsoft\Windows\CurrentVersion\Uninstall\com.xinjing.desktop
```

（如存在 `Uninstall\com.xinjing.desktop` 之外的同类项，一并删除。）

> 可选：用 PowerShell 一步删除（在你自己机器上运行，需确认路径）：
> ```powershell
> Remove-Item "HKCU:\Software\Microsoft\Windows\CurrentVersion\Uninstall\com.xinjing.desktop" -Recurse -Force -ErrorAction SilentlyContinue
> ```

### 步骤 4：以管理员身份运行安装器
右键步骤 2 复制出来的 `xinjing-setup-3.6.8.exe` → **以管理员身份运行**。
- 此时注册表已无卸载项 → 安装器判定为全新安装，直接复制文件，**不再调用旧卸载器**；
- 正常走完即升级到 3.6.8。

## 五、针对未来的加固建议（3.6.9 候选）

要让后续自动更新**自愈**、不再被旧卸载器 abort 卡死，可在 `build/installer.nsh` 中增强，但存在 NSIS 覆盖限制需注意：

- `uninstallOldVersion` / `handleUninstallResult` 是 `Function`（非宏），electron-builder 无条件定义；我的 `build/installer.nsh` 被 `!include` 在头部，若再定义同名 Function 会与 electron-builder 的后续定义**重名编译报错**，故不能直接覆盖。
- `customCheckAppRunning` 之所以能覆盖，是因为它对应 `!ifmacrodef` 守卫的宏。卸载步骤**没有同等宏守卫**，故更稳妥的长期方案是：
  1. 在 `customCheckAppRunning`（或新宏）里，于安装开始前先 `RMDir /r` 兜底清理旧目录（配合管理员权限），降低旧卸载器 abort 概率；或
  2. 升级 electron-builder 版本，关注其是否修复了"卸载器 abort 后安装器硬失败"的行为；或
  3. 发版说明里固化"遇到本错误 → 走免安装版 / 手动清注册表"的标准自救路径（即本文档）。

> 当前（3.6.8）未做此加固——因本错误属升级链路，且手动清注册表已可 100% 绕过。是否纳入 3.6.9 待你确认。

---

## 六、一句话总结

`:2` = 旧卸载器删不掉旧目录而 abort；不是心镜进程在跑（你已确认没有），而是 Defender/Explorer 等外部进程锁了旧安装目录里的文件。把**旧安装目录 + 卸载注册表项**都清掉，再用管理员身份跑 `xinjing-setup-3.6.8.exe` 做全新安装即可；或永远用免安装版规避。
