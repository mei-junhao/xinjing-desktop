## 三、核心问题（按严重度）

### 3.1 产品主干偏航：8 项需求达成度低

| # | 需求 | 状态 | 客户经理判读 |
|---|------|------|--------------|
| ① | 云激活（WS-A） | 调研完、服务端未开发 | 付费闸门未真正闭环，当前"激活=全功能"二元门控与对外"会员制"叙事不匹配 |
| ② | 真人督导上传转写→生成记录 | 一期已交付；音视频转写未做 | 可用，但只支持文字稿 |

| ④ | 帮助文档 | **空白** | 与"隐私优先、授权透明"主张直接冲突，用户上手无引导 |

**结论**：8 项中②⑤交付、①③部分交付且留待拍板项、④⑦⑧基本空白。咨询日历 / 授权分层 / 云激活产品化三大里程碑被持续后移——用户侧看不到路线图收敛，易产生"做了很多 Agent 花活，但我要的个案管理工作台主干没补齐"的落差。

### 3.2 记账"流程割裂"（竞品分析的核心结论）

《记账模块竞品分析》原话：**"心镜记账模块最大的问题不是缺功能，是流程割裂。咨询师做完 session 写笔记是一件事，记费是另一件事，但心镜把它们放在了两个页面。"** 五个竞品共同教训：**"把计费嵌入到 session 闭环中，比做一个功能更全的独立记账模块，价值大十倍。"**

具体别扭（来自《记账界面修改意见》+ 三份演示 HTML 印证）：
- 布局割裂：真代码 `billing.html` 单列长滚动，记一次/看明细/回访在三个位置来回切；`renderAll()` 全量重绘会闪、丢焦点。
- 数据口径硬伤："总收入"含未缴虚高、"小时数"实为次数；修改意见要求拆应收/已收。
- 每次录入都在摩擦：单价 readonly 无法单次改价、改记录用原生 `prompt()`、删记录 `confirm()` 硬删无撤销。
- 账单能力缺失：有 CSV 导出但"CSV 是给机器读的，账单是给来访者看的"，咨询师每月对账心镜帮不上忙。
- 首页不回答"钱在哪"：竞品（Quitado 三大数字、SimplePractice 仪表盘）把财务摘要放第一屏，心镜的统计要主动切 Tab 且只有数字无图。

**→ 这是性价比最高的改造：把"做完 session 到记完费"从 4 步砍到 1 步。**（见 HTML 示意 ④⑤）

### 3.3 "功能宣发强、实测弱"的稳定性叙事

- **DeepSeek 连环 400**：v1.5.1（工具名含点号）、v1.5.2（多轮截断致 tool 消息孤儿）、v1.5.3（发送边界配对）连修三版才稳。最典型的"宣发可用、实测踩坑"。
- **档位曾谎报**：v1.4.1 前即使 API 接入失败，横幅仍显示"已接入高性能模型"，AI 也跟着撒谎（`getTier()` 只看密钥非空不看能否连通），修复后才诚实化。
- 客户经理处置：**稳定性/兼容性应列为"对外承诺"前的第一优先**，任何"已支持某模型"的宣称都要有实测背书。

### 3.4 商业化未闭环

- 云激活：**调研完成、服务端未开发**（《云激活实现方案》明确"尚未开发"）。当前付费闸门可能仍未真正闭环。
- 会员/分层：Free/Pro/Custom 仅停留在《下一步计划》设计稿，无 entitlement 实现、无授权文档。
- 付费价值不清：v1.6.0 的"数据感知/跟进提示"恰恰最该放进 Pro 档的差异化能力，却尚未与授权模型挂钩——"为什么付费、付了得什么"用户（和未来客户）无从得知。

### 3.5 可靠性与信任风险（最伤口碑）

数据持久化高度依赖**隐含契约**（固定端口、API 级迁移、构建期版本），一旦打破即以高危形式暴露：
- **数据"丢失"假象**：早期随机端口 → origin 变 → 每次读空库（《数据丢失 bug 诊断》）。真因数据没丢，是"被切成多个端口库"。
- **版本号失真**：设置页"关于"写死旧版本号，桥接失效即静默回退（《版本号 bug 分析》）。对外版本号不可信直接损信任。
- **中文输入失效**：因 `style.css` 缺 `.hidden` 类，锁提示常驻干扰 Windows IME 组合输入（《设置页无法输入汉字》）。一行 CSS 缺失造成"打字不能用"的严重事故。
- **处置**：这类问题优先级应**高于体验优化**——"客户数据不丢、版本号真实、能正常打字"是底线。

### 3.6 移动端与日历空白

- **Android**：有清晰方案（Capacitor + WebView，约 33 天，零服务器），但**未落地**；FAQ 仍写"手机不能用"。对外暂不宜承诺移动端。
- **咨询日历**：代码库零文件，排期 #21–#26 全 ◻️。这是"个案管理工作台"主干，缺失影响最大。

### 3.7 文档/帮助缺失

帮助文档（需求④）、授权能力文档（需求⑦）两项完全空白。与"隐私优先、授权透明"主张冲突，也影响新用户留存与付费转化。

## 四、客户经理视角的优先级建议

> 原则：**先修信任（数据安全/版本/输入），再补主干（日历/会话计费闭环/账单），最后谈增长（会员/移动/AI 花活）。**

### P0 — 信任底线（立刻，对外承诺前必须）
1. 数据持久化回归测试常态化（固定端口契约写进 CI 断言）。
2. 版本号构建期 stamping（`version.generated.js`），杜绝写死旧版本。
3. 全站补齐缺失基础 CSS 类（如 `.hidden`），并加 IME 输入冒烟测试。
4. 模型兼容矩阵实测背书（DeepSeek/硅基/OpenAI/Kimi/智谱/通义/豆包）后再宣称"支持"。

### P1 — 产品主干（本季度，兑现承诺）
5. **会话→计费闭环**：写完 SOAP 弹轻量计费卡（自动带日期+来访者+默认单价），把"记一次"从 4 步砍到 1 步（呼应竞品分析 P0-1）。
6. **首页回答"钱在哪"**：工作台首屏 4 卡（本月应收/已收/待收 + 待收来访者数）+ 今日待记列表 + 收入趋势图（呼应竞品分析规律 3）。
7. **可外发账单 + 趋势图**：来访者账单 PDF/打印 HTML + 月度收入趋势 + 来访者收入排行（已在 `billing-redesign.html` 示意）。
8. **咨询日历**：从规划补到可用（来访者关联、周期、增删、智能提醒），这是 8 项需求里最该补的主干。
9. **帮助文档 + 授权说明**：补需求④⑦，把"隐私/授权/会员"讲清楚，提升转化。

### P2 — 增长与精细化（下阶段）
10. 云激活服务端 + 会员 entitlement 闭环，让"60 天试用→会员"真正可收费。
11. Agent 浮窗升级（呼吸球/拖动/全屏，v2.0 #6）、主动提醒引擎（同周同时提示增加、多次取消询问）。
12. Android 落地（Capacitor 复用 90% 前端），打开移动叙事。
13. 报告中心资产化（报告历史列表、量表沉淀），而非一次性动作。

## 六、对外叙事建议（客户经理话术）

**该讲**：
- "你的来访者数据 100% 留在本机，默认不离开你的电脑。"（隐私壁垒）

**对内给研发的反馈排序**：信任底线（P0）> 会话计费闭环 + 首页财务摘要 + 账单趋势（P1）> 会员/移动/AI 花活（P2）。先让"咨询师每天真的用得顺"，再谈"卖得更贵"。
