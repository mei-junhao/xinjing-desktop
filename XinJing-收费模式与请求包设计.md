# 心镜 XinJing · 收费模式与请求包设计

> 版本：草案 v1（2026-07-13）
> 关联：`XinJing-v1.7.0-微信支付云激活技术方案.md`（本方案是其上游产品决策）
> 状态：待用户拍板定价与档位边界；技术落点已对齐真实代码。

---

## 摘要

本设计将心镜的收费拆为**两个正交维度**，避免把"功能权限"与"AI 算力来源"混为一谈：

- **维度 A — 功能权限档位**（free / pro / custom）：决定用户能用哪些功能。
- **维度 B — AI 算力来源**（BYOK 自带密钥 / 官方请求包 / 内置免费兜底）：决定谁为大模型调用付费、如何计量。

两者正交组合：一个 pro 用户既可自带 API 密钥（BYOK，你不计费），也可购买你的官方请求包（走代理计量扣费）。请求包是**独立增值商品**，面向"不会配置 API 或不愿自付大模型费用"的用户，任何档位均可购买。

关键前置结论（基于代码实证）：**"限制请求次数 + 提供 API + 卖请求包"三项，技术上都要求相关请求改走韩国服务器代理（计量网关），由服务端持有真密钥、鉴权、计数、扣包。BYOK 请求维持客户端直连。** 当前内置模型密钥明文硬编码在客户端（`ai.js` L24），存在被反编译盗刷的商业漏洞，须一并迁移至代理。

## 2. 维度 A：三档功能权限

设计原则：档位差异必须**代码可门控且用户可感知**，避免"仅 logo 色差"的空档位。

### 2.1 Free（免费版）

| 项 | 权限 |
|---|---|
| 基础个案管理（来访者 / 会谈 / 记账 / 报告） | **永久免费，不限量**（移除现有 5 来访者 / 50 督导 cap） |
| AI 功能（助手 / 督导 / 大师圆桌 / Agent） | **60 天试用**（`AI_TRIAL_DAYS=60`），仅走**内置免费兜底模型**（低性能、限速、每日限量） |
| 60 天后 | AI 锁定；基础功能继续免费；可通过"买请求包"或"升级 pro"恢复 AI |
| 数据导出 | 带水印 / 受限 |

### 2.2 Pro（标准版）

| 项 | 权限 |
|---|---|
| 全部 AI 功能 | 助手 / 督导 / **大师圆桌** / Agent 全解锁 |
| 私有模型接入（BYOK 自配） | ✅ 归 pro（用户自带 key 直连，你不计费）|
| 数据导出 / 水印解除 | ✅ |
| AI 算力来源 | 可 **BYOK**（自带密钥直连，你不计费）；或使用**官方请求包**（走代理扣量）；**每月赠送基础额度**（如 N 次，具体见 §5） |

### 2.3 Custom（旗舰版）

| 项 | 权限（在 pro 基础上叠加）|
|---|---|
| 官方请求额度 | **每月赠送额度更高** + **请求包单价更低**（阶梯优惠） |
| 模型档位 | 可选**更高性能模型 / 更高并发**（如从 DeepSeek-V3 升至更强模型，由代理侧路由） |
| 优先技术支持 / 定制开发 | ✅（服务类，UI 标识 + 专属入口） |
| 自定义督导（开发者专属制作） | ✅ **custom 独占**。由开发者为 custom 用户制作专属督导人格（非用户自编辑提示词），如特定取向督导师、私有人设；`supervisor` 列表以 `customUnlocked` 解锁，pro 不可见 |

> 三档实质差异总结：**Free = 基础免费 + AI 限时限量兜底；Pro = 全功能解锁 + 基础官方额度 + 可 BYOK；Custom = Pro + 更高额度/更优单价/更强模型/自定义督导（开发者制作）+ 专属服务。**

## 3. 维度 B：三种 AI 算力来源

这是本次新增需求的核心。三种来源在客户端 `activeSource()` 中表达为三态：

| 来源 | 谁付大模型费 | 是否走代理 | 是否计量 | 面向人群 |
|---|---|---|---|---|
| **BYOK（自带密钥）** | 用户自付（充值到 DeepSeek/OpenAI 等） | 否，客户端直连 | 否 | 会配置、有自己 key 的技术用户（pro/custom） |
| **官方请求包（Managed）** | 用户向你购买请求包 | **是，走韩国代理** | **是，按次/token 扣减** | 不会配 key、不愿自付大模型费的用户（任意档位可买） |
| **内置免费兜底** | 你承担（低成本小模型） | **应改为走代理**（防盗刷） | 是，每日限量 | Free 用户试用期 + 兜底 |

### 3.1 关键改造：内置兜底必须迁移到代理

当前内置模型密钥明文在客户端，等于把你的钱包挂在公网。改造后：客户端不再持有任何真密钥，"内置免费"和"官方请求包"都以 `machineCode + license` 向代理鉴权，代理持真密钥转发并计数。BYOK 是唯一仍在客户端持密钥的来源（那是用户自己的 key，风险自负）。

## 4. 请求包机制（增值商品）

### 4.1 计量单位

两种口径，各有取舍：

- **按"次"计费**（推荐面向用户）：直观易懂（"100 次对话包"）。缺点：单次 token 消耗差异大（一次长督导 vs 一句问答），成本不均。
- **按 token 计费**（推荐内部核算）：公平贴合成本。缺点：用户难感知。

> 建议：**对用户展示"次数"，内部按 token 折算**。例如 1 "次" = 封顶 4K token，超出的长对话按 2 次计。兼顾易懂与成本可控。

### 4.2 额度类型

| 类型 | 说明 | 归属 |
|---|---|---|
| 赠送额度（quota） | 每月随会员发放，**月度清零不累积** | pro/custom 会员权益 |
| 购买额度（credit） | 请求包购买所得，**永久有效或长有效期，可累积** | 任意档位可买 |

扣减顺序建议：**先扣赠送额度（当月清零，不用白不用），再扣购买额度**。

### 4.3 限速与防滥用

- 每分钟 / 每日请求上限（防脚本刷包）。
- 单请求 token 上限（防超长上下文击穿成本）。
- 余额为 0 时：官方来源降级为"请引导 BYOK 或购买"，不静默失败。

### 4.4 购买链路（复用 v1.7.0 微信支付通道）

请求包购买**完全复用** v1.7.0 已设计的微信 Native 扫码支付 + Webhook 入库通道，仅新增商品类型：

```
用户点"购买请求包" → 客户端请求下单（tier无关，商品=pack_N）
  → 服务端调微信 Native 下单，返回 code_url → 客户端展示二维码
  → 用户扫码支付 → 微信 Webhook 回调韩国服务器（验签+解密）
  → 服务端 credits 表：machineCode 余额 += 包内次数
  → 客户端下次调用/心跳时读到新余额
```

## 5. 交叉矩阵：档位 × 算力来源

| | 内置免费兜底 | 官方请求包 | BYOK |
|---|---|---|---|
| **Free** | ✅ 前 60 天限量 | ✅ 可购买 | ✅ 可自配（但 AI 功能须在试用期内或已购包）|
| **Pro** | —（已有更好来源）| ✅ 每月赠 N 次 + 可加购 | ✅ 无限量直连 |
| **Custom** | — | ✅ 每月赠更多 + 加购单价更低 + 可选更强模型 | ✅ 无限量直连 |

> 读法：**BYOK 是"能力"，请求包是"商品"，档位是"资格 + 优惠力度"**。三者独立又叠加。

## 6. 定价结构（区间建议，最终由你拍板）

> 以下为**结构建议**，非最终定价。定价需结合你的大模型成本（DeepSeek 约 ¥1–2/百万 token）、目标人群付费力综合决定。

| 商品 | 建议结构 | 备注 |
|---|---|---|
| Pro 会员 | 月付 / 年付（年付 8 折）| 含每月赠送额度 |
| Custom 会员 | Pro 的 2–3× | 含更高赠额 + 更优包单价 + 强模型 |
| 请求包 | 阶梯装（如 100 / 500 / 2000 次），**买多单价递减** | 任意档位可买；custom 享额外折扣 |

**成本护栏**：内置兜底与赠送额度是你的净成本，须设每日/每月硬上限，避免个别用户拖垮毛利。建议内置兜底日限（如 20 次/日）足够体验、不足以替代付费。

## 7. 技术落地改造清单

按改动量与依赖排序（可小步实施，每步可回滚）：

### 7.1 服务端（韩国服务器，最重）

1. **代理升级为计量网关**：`server.js` 增加 `POST /v1/chat`（或复用现路径 + 鉴权头），校验 `machineCode + license` → 查 `credits`/`quota` 余额 → 足额则持真密钥转发大模型、扣减、记流水；不足则拒绝并返回引导码。
2. **多后端路由**：按 tier / 商品选择大模型后端（DeepSeek 基础档、更强模型给 custom）。
3. **额度表**：`quota`（月度赠送，随 license 发放/清零）+ `credits`（购买累积），与 v1.7.0 `licenses`/`orders` 同库（SQLite）。
4. **请求包商品接入微信支付**：Webhook 新增 `pack_*` 商品类型 → 到账加 credits。
5. **限速中间件**：每分钟/每日/单请求 token 上限。

### 7.2 客户端（`D:\xinjing-electron`）

6. `ai.js`：`activeSource()` 新增 `'managed'` 三态；`resolveConfig` 增加"走代理"分支（`baseUrl = xinjingchat.online`，鉴权头带 machineCode/license，不再带真大模型 key）。
7. `ai.js`：**移除硬编码 SiliconFlow 密钥**，内置兜底改为经代理调用（闭合盗刷漏洞）。
8. 额度 UI：设置页/AI 面板显示剩余次数、"购买请求包"入口、"配置自己的 API（BYOK）"入口并列。
9. `license-core.js`：档位常量 + `AI_TRIAL_DAYS=60`（与 v1.7.0 §8 决策一致）。
10. custom 门控：`ai.js` 按 tier 的额度/模型分支。

### 7.3 依赖关系

本设计的"官方请求包"依赖 v1.7.0 的**云激活/付费真相源**先落地（licenses 表 + 微信支付 + machineCode 鉴权）。**建议顺序**：先完成 v1.7.0（授权 + 支付底座）→ 再在其上加"计量网关 + 请求包"（可作为 v1.8.0）。

## 8. 关键风险与待决策项

| 项 | 类型 | 说明 |
|---|---|---|
| 硬编码密钥盗刷 | **P0 安全** | `ai.js` L24 明文 key 须尽快迁移至代理，即使收费模式未上线也应先修 |
| 计量口径 | 待拍板 | 次数 vs token（建议展示次数、内部 token） |
| 赠送额度是否累积 | 待拍板 | 建议月度清零（控成本），购买额度累积 |
| 内置兜底日限 | 待拍板 | 建议 ~20 次/日，够体验不够替代付费 |
| Custom 独占模型 | 待拍板 | 是否为 custom 接入更强/更贵模型（增成本，也是差异化卖点） |
| 定价 | 待拍板 | 见 §6，需结合成本核算 |

## 9. 一句话总结

**把"功能档位（free/pro/custom）"和"算力来源（BYOK / 官方请求包 / 内置兜底）"拆成两个正交维度：档位卖资格与优惠，请求包卖用量，BYOK 让技术用户自付。三者都要求相关请求走韩国代理计量——你的服务器已具备基础设施，核心改造是把代理升级为计量网关，并先堵上客户端硬编码密钥这个盗刷漏洞。**
