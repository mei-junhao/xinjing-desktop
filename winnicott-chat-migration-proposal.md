## 一、结论先行

迁移在技术上可行，但原方案中的“复制五个 HTML 页面，再替换 API 和存储”不应执行。

复核源码后，推荐路线是：

**以心镜现有 `masters.html + masters.js + masters-core.js + ai.js + Store` 为唯一运行内核，按能力吸收 winnicott-chat 的内容与体验，不迁入第二套页面、第二套 AI 路由和第二套存储。**

原因如下：

1. 心镜已经具备大师列表、一对一、圆桌、长期摘要、IndexedDB 历史、知识库内置、温度与详细度控制、导出等核心能力，不是从零迁移。
2. winnicott-chat 的五个核心页面合计约 47 万字节、7,784 行，CSS、状态、API、历史逻辑大量内联和重复。直接复制会形成两套对话内核。
3. 网页端直接 `fetch`、SSE 流式解析、`localStorage` API Key、远程脚本、公开代理和 Electron 的安全边界不同，不能按“纯前端完全兼容”处理。
4. 两个项目的知识库大部分已经同源；真正缺口集中在科胡特、温尼科特知识版本冲突、流式输出、分享卡片、章节化和局部 i18n。
5. 心镜已有独立督导模块和统一授权体系。继续搬入 `ai-supervisor.html`、`consultant-a.html` 会造成业务重叠和权限绕行。

因此，本项目应定义为“能力收敛与补齐”，不是“网页项目整体嵌入”。

### 2.4 对原方案关键结论的修正

| 原结论 | 复核结论 |
|---|---|
| “纯前端代码可直接运行在 Electron 中” | 语法可运行不等于可安全集成。网络、窗口、密钥、文件来源和权限模型均需适配。 |
| “核心工作只有 AI、鉴权、存储三项” | 还包括流式协议、内容版本、统一内核、数据迁移、并发状态、安全、测试与发布兼容。 |
| “迁入四层降级可提升稳定性” | 不应让功能页决定供应商。统一 AI 层才是可维护的稳定性来源。 |
| “知识库离线，因此断网仍可对话” | 断网时只能查看本地知识和历史；模型推理仍需网络，除非未来接入本地模型。 |
| “使用相对路径读取知识库即可” | Electron `file://`、打包路径和 CSP 下运行时 fetch 容易出问题。应继续构建期内置。 |
| “新增几个入口按钮即可完成导航” | 心镜已有统一入口。应直接增强现有 `masters.html`，避免出现新旧两套入口。 |

## 三、技术路线选择

### 3.1 三种路线对比

| 路线 | 做法 | 优点 | 主要问题 | 结论 |
|---|---|---|---|---|
| A. 直接复制页面 | 将五个 HTML 放入 `app/masters/` | 初期可快速看到页面 | 双内核、重复存储、权限绕行、远程资源、安全与维护成本高 | 拒绝 |
| B. iframe 或独立窗口嵌入 | Electron 内加载网页或本地页面 | 与网页外观最接近 | 跨 origin 数据不通、导航割裂、授权难统一、离线与更新不可控 | 拒绝 |
| C. 能力收敛 | 保留心镜内核，逐项吸收源码能力 | 单一事实源、数据一致、权限一致、可持续维护 | 前期需整理核心边界 | 推荐 |

### 3.2 推荐目标架构

```text
masters.html
  └─ masters.js                 仅负责 DOM、事件、可访问性、页面状态
       └─ masters-core.js       唯一对话领域内核
            ├─ masters-data.js  大师元数据与人格入口
            ├─ Knowledge        构建期内置知识内容
            ├─ AI.send()        唯一模型调用入口
            └─ Store            唯一持久化入口

可选能力模块：
  master-share.js               分享卡片与本地导出
  master-i18n.js                大师模块文案字典
  master-content-manifest.json  内容来源、版本和哈希清单
```

核心约束：

1. `masters.js` 不再复制 prompt 拼装、历史截断和圆桌编排逻辑。
2. `masters-core.js` 成为页面、Agent 工具和后续入口共同使用的唯一领域服务。
3. 所有模型请求只经过 `AI.send()`；大师模块不保存 API Key，不知道具体供应商地址。
4. 所有对话只经过 `Store.saveMasterConversation()`；localStorage 只允许保存非敏感 UI 偏好。
5. 所有知识文件以心镜仓库为运行时事实源，构建时生成 `knowledge.builtins.js`。

## 四、具体实施设计

### 4.1 第一步：先统一心镜内部的双路径

当前 `masters-core.js` 已声明为纯领域内核，但 `masters.js` 仍保留一套完整的 prompt、圆桌和 AI 调用实现。Agent 工具与页面路径因此可能产生不同结果。

实施时应先完成：

1. 将一对一 prompt 构建、知识选择、上下文截断、摘要触发、圆桌分轮和调用参数全部收口到 `masters-core.js`。
2. `masters.js` 只传入用户动作，接收领域事件并渲染。
3. Agent 工具调用同一个 `masters-core.js`，不再自行拼 prompt。
4. 对现有行为建立字符级或结构级回归测试，确保收口前后输出请求结构一致。

这是后续所有迁移的前置条件。若不先收口，新增流式、科胡特和章节能力会继续被复制两次。

### 4.2 大师内容与知识库同步

不建议人工反复复制整个 `public/`。应建立显式内容清单：

```json
{
  "source": "winnicott-chat v5.2",
  "approvedAt": "YYYY-MM-DD",
  "masters": {
    "kohut": {
      "knowledge": "sha256:...",
      "perspective": "sha256:..."
    }
  }
}
```

同步规则：

1. 仅同步允许清单内的 `*-knowledge.md`、`*-perspective.md` 和经批准的人格素材。
2. 共同文件先比较 SHA-256；相同则不改，不同则进入人工审阅。
3. `winnicott-knowledge.md` 当前两边哈希和大小不同，必须三方审阅差异，不能用网页版直接覆盖心镜版。
4. 新增科胡特时同时更新 `masters-data.js`、知识生成脚本中的 `known_keys`、构建产物和测试。
5. `winnicott-knowledge-full.md` 体积约 18 万字节，不应整篇注入模型。只有在完成上下文预算设计后才能纳入。
6. `cangjie-perspective-full.md`、`nvwa-perspective-full.md` 属于督导素材，不自动视为大师对话知识。

知识选择策略：

- 低温度：情感/抱持风格片段。
- 中温度：人格视角和核心理论摘要。
- 高温度：核心人格 + 与当前问题最相关的知识片段。
- 始终保留人格铁律、用户当前问题和最近对话；知识内容不得挤掉最近上下文。

第一版可使用本地标题与关键词评分选择片段。不要把仓库里尚未接线的 RAG WIP 直接作为本迁移依赖；RAG 应另立方案和验收。

### 4.3 AI 调用与流式输出

网页端的价值之一是逐字流式显示。心镜当前 `AI.send(messages, callback, options)` 返回完整结果。推荐保持旧签名兼容，增加可选能力：

```javascript
AI.send(messages, callback, {
  temperature: 0.6,
  maxTokens: 1024,
  onDelta: function (text) {},
  signal: abortController.signal
});
```

实现要求：

1. 未传 `onDelta` 时保持现有非流式行为，避免影响督导、报告和工具调用。
2. 传入 `onDelta` 时由 `ai.js` 统一解析 OpenAI 兼容 SSE，不允许业务页自己解析。
3. 支持 `AbortSignal`，用户取消、切换对话或关闭页面时终止请求。
4. 首个 token 出现前失败，可以按心镜策略降级并重试。
5. 已输出部分内容后失败，不得静默切换模型重放，否则会出现重复或语义断裂。应将消息标为“已中断”，允许用户显式重试。
6. 流式中不逐 token 写 IndexedDB。采用内存更新与节流快照，结束、中断或离开页面时落盘。
7. 工具调用路径默认保持非流式，直到工具增量协议被单独验证。

圆桌模式第一阶段不需要多路 token 同时滚动。保持当前“每位大师完成后落一条消息”的方式更稳定；只为一对一模式增加流式输出。

### 4.4 统一数据模型与兼容迁移

建议将现有会话模型升级为显式版本，但保持旧字段可读：

```javascript
{
  schemaVersion: 2,
  id: "mc_...",
  mode: "1v1" | "round",
  masterKeys: ["winnicott"],
  title: "与温尼科特对话",
  messages: [
    {
      id: "msg_...",
      role: "user" | "assistant" | "system",
      masterKey: "winnicott",
      content: "...",
      status: "complete" | "streaming" | "interrupted" | "error",
      createdAt: "..."
    }
  ],
  summary: "...",
  settings: {
    temperature: 60,
    detail: 50,
    locale: "zh-CN"
  },
  sourceVersion: "winnicott-chat-v5.2",
  createdAt: "...",
  updatedAt: "..."
}
```

迁移规则：

1. 旧会话无 `schemaVersion` 时按 v1 读取，首次保存时惰性升级。
2. 历史中出现 `roundtable` 时归一为 `round`，但读取期兼容两者。
3. 旧消息缺少 id、状态和时间时补默认值，不修改原 content。
4. 升级前保留一次 IndexedDB 备份，失败则继续读取 v1，不删除旧数据。
5. 网页版 localStorage 与 Electron 的 IndexedDB 不同 origin，无法可靠自动读取。若需要迁移网页历史，应增加“网页导出 JSON -> 心镜显式导入”流程，并展示导入预览与冲突处理。
6. API Key、公开代理配置、打赏计数、访问统计等网页设置不得导入。

### 4.5 分享、章节和导出

分享卡片：

- 不动态加载 CDN `html2canvas`。
- 优先使用本地 Canvas 或随包依赖生成 PNG。
- 所有模型文本按纯文本绘制，统一换行、最大长度和脱敏预览。
- 默认不包含来访者姓名、联系方式、个案编号和账务信息。
- 导出前由用户确认选中的消息范围。

章节化：

- 章节标记作为会话派生数据，不改写消息正文。
- 第一版采用规则触发：话题间隔、用户显式新议题、消息数量阈值。
- AI 自动命名章节属于可选增强，失败不影响对话保存。
- 章节标题也应存入本地，不上传额外上下文，除非需要 AI 命名且用户已授权调用。

导出：

- 保留现有 Word 导出。
- 增加 Markdown/JSON 时使用心镜统一下载能力。
- JSON 导出包含 `schemaVersion`，用于未来可逆导入。

### 4.6 i18n 与主题

不迁入网页端整页样式和每页暗色开关。

1. 大师模块使用心镜全局 `xj_theme` 和设计 token。
2. 将 `masters.html`、`masters.js` 中用户可见文案抽为 key。
3. 第一阶段只保证 `zh-CN`；第二阶段接入 `en-US`，复用网页端字典内容但不复用其 DOM 操作代码。
4. 大师人格的输出语言由 prompt 明确控制，界面语言和模型回答语言分开配置。
5. 大师专属颜色只能作为局部识别色，不能覆盖全局可访问性和深色主题对比度。

### 4.7 督导、来访者与临床数据

`ai-supervisor.html` 不作为新页面迁入。可复用的是方法论素材和局部交互，不是其存储与 API 代码。

与临床数据打通必须遵循：

1. 大师对话默认是独立学习空间，不自动读取任何来访者资料。
2. 用户显式选择来访者或会谈后，才允许注入限定范围的脱敏材料。
3. 对话结果保存为咨询记录或督导记录前必须预览和确认。
4. 关联字段使用可选 `clientId`、`sessionId`，删除关联对象时不删除原大师对话。
5. 写入督导走现有 `Store.saveAiSupervision()` 或正式业务 API，不写自建 localStorage。
6. 账务能力不在大师对话中隐式触发，不修改 `billing-shell.html` 的 `billableSessionsFor` / `billableSessions` 隔离逻辑。

`consultant-a.html` 当前带独立业务身份和历史密码门。除非产品层明确其身份、授权、免责声明和售卖方式，否则只保留内容评估，不进入正式运行时。

### 4.8 Electron 安全边界

迁移后应满足：

- 大师模块零远程 JavaScript。
- 大师模块零明文 API Key，零 per-master Key。
- 大师模块零直接模型 `fetch`，统一经过 `AI.send()`。
- 外部链接统一经过 preload IPC，并在主进程校验协议和域名白名单。
- 模型输出继续通过 `App.escapeHtml()` 或纯文本节点渲染。
- 本地知识内容只从构建产物读取，不接受运行时任意路径。
- 不迁入 busuanzi、远程打赏脚本、网页留言板和公开代理健康检测。

当前主进程 `xj:openExternal` 对 URL 校验较宽。若本次增加分享或外链入口，应把协议限制为 `https:`、`mailto:` 等明确集合，并对固定产品域名使用白名单。该项属于安全前置，不是视觉优化。

## 五、主要难点与应对

### 难点 1：心镜自身存在页面路径与纯核心路径重复

影响：同一大师在页面和 Agent 中可能使用不同 prompt、知识和截断规则。

应对：先将 `masters-core.js` 设为唯一领域内核，再迁能力。该步骤不可跳过。

### 难点 2：流式输出与自动降级存在语义冲突

影响：输出一半后换模型会重复、改口或破坏历史。

应对：只允许首 token 前自动降级；首 token 后失败标记中断，由用户决定重试。

### 难点 3：知识库不是越长越好

影响：温尼科特完整知识和 skill 合并后可能远超有效上下文，增加成本并稀释人格。

应对：建立上下文预算、内容分段和相关片段选择；人格核心永远优先于扩展知识。

### 难点 4：两边内容已产生分叉

影响：简单覆盖会丢失心镜修订，简单忽略会漏掉网页端新内容。

应对：哈希清单 + 人工三方合并 + 内容回归样例。首先处理温尼科特冲突，再新增科胡特。

### 难点 5：圆桌并发、成本和故障隔离

影响：一次用户输入可能触发多轮、多模型请求；任一失败不能拖垮整场。

应对：限制参与人数和轮数；每位大师独立超时；结果按大师落盘；失败只显示在对应发言位。

### 难点 6：流式状态与 IndexedDB 写入频率

影响：逐 token 持久化产生大量写操作，崩溃恢复又要求保留部分内容。

应对：内存渲染、500-1000ms 节流快照、结束状态强制落盘；消息状态显式记录。

### 难点 7：网页历史无法直接跨 origin 读取

影响：用户误以为安装桌面版后网页历史会自动出现。

应对：提供显式 JSON 导出/导入，而不是复制 Chromium LevelDB 或读取网页 origin。

### 难点 8：临床数据边界

影响：大师对话若自动读取个案资料，会扩大隐私暴露和误写风险。

应对：默认隔离、显式选择、最小字段、脱敏预览、写入确认、操作审计。

### 难点 9：大单页源码难以直接测试

影响：从网页页中复制函数会同时复制隐式 DOM、全局变量和状态依赖。

应对：只提炼行为和内容；先写契约测试，再在心镜模块化实现，不复制整段页面脚本。

## 六、分阶段实施计划

### 阶段 0：范围冻结与基线（1-2 人日）

- 冻结源版本、心镜目标版本和内容哈希。
- 建立功能矩阵：保留、补齐、舍弃、另立项目。
- 保存现有大师对话数据样本和 prompt 请求快照。
- 明确本期不包含打包、发布、托盘、置顶和全局快捷键。

验收：所有迁移对象有唯一来源和唯一归属，无“顺手复制”的文件。

### 阶段 1：统一领域内核（3-5 人日）

- 将 `masters.js` 的领域逻辑收口到 `masters-core.js`。
- 页面和 Agent 使用同一 prompt、知识选择和圆桌编排。
- 建立会话 schema v2 的兼容读取与惰性升级设计。
- 保持现有 UI 和用户数据不变。

验收：同输入、同配置下，页面与 Agent 生成的模型请求结构一致。

### 阶段 2：内容同步与科胡特（2-3 人日）

- 建立内容 manifest 和哈希校验。
- 人工合并温尼科特知识冲突。
- 增加科胡特配置、知识、视角和测试。
- 更新知识构建脚本，不手改生成文件。

验收：12 位大师均有配置、知识选择和最小对话测试；生成脚本无未知 key 警告。

### 阶段 3：一对一流式与取消（4-6 人日）

- 扩展 `AI.send()` 的 `onDelta`、`signal` 和中断状态。
- 一对一页面接入流式渲染、取消和显式重试。
- 实现首 token 前降级、首 token 后中断规则。
- 加入节流保存和页面离开清理。

验收：取消无残留请求；中断无重复文本；旧的非流式调用全部兼容。

### 阶段 4：分享、章节、JSON 导入导出（4-6 人日）

- 本地分享卡片生成，不加载 CDN。
- 章节作为派生数据接入。
- 增加 schema 化 JSON 导入导出与冲突预览。
- 保留并回归 Word 导出。

验收：分享内容不含未选择的消息和临床身份字段；导出后可无损导入测试库。

### 阶段 5：i18n 与临床关联（3-5 人日，可拆期）

- 大师模块文案 key 化，接入 `zh-CN` / `en-US`。
- 增加可选 client/session 关联。
- 保存到咨询记录或督导记录前预览确认。

验收：默认对话不读取来访者数据；任何临床写操作都有明确用户确认。

### 阶段 6：回归与发布前审查（3-5 人日）

- 单元测试：prompt、知识选择、schema 迁移、SSE 解析、降级状态机。
- 集成测试：AI 回调兼容、IndexedDB、授权门禁、导入导出。
- UI 测试：一对一、圆桌、取消、重试、历史、深色模式、窄窗口。
- 安全审查：远程脚本、直接 fetch、密钥、外链、输出转义。
- 运行心镜既有自测和全量 HTML 结构检查。

验收：全部阻断项清零后才允许进入构建与发布流程；本方案本身不执行构建与发布。

预计总量：20-32 人日。若只完成统一内核、科胡特和一对一流式，预计 9-14 人日。

## 七、明确不迁入的内容

以下内容不应随本项目进入心镜运行时：

- `backup-*`、`index-dev*`、`v4-*` 等备份和遗留页面。
- 网页首页、留言板、busuanzi 访问统计。
- 远程动态加载的 `html2canvas`。
- 每位大师独立 API 地址、API Key 和线路健康检测。
- 网页公开代理和 SCF 地址的业务页直连逻辑。
- 打赏弹窗和网页作者推广入口。
- `consultant-a.html` 的明文密码门。
- 新建第二套 AI 督导页面和第二套督导历史。
- 与迁移无关的系统托盘、窗口置顶、全局快捷键。

这些功能不是“漏迁”，而是与心镜桌面端架构、安全或产品定位冲突。

## 八、验收标准

### 8.1 架构验收

- 大师页面与 Agent 共用 `masters-core.js`。
- 大师模块不存在模型直连和 API Key 存储。
- 大师对话只写 `Store.masterConversations`。
- 知识库只从批准的构建产物读取。
- 旧会话无需人工处理即可继续打开和保存。

### 8.2 功能验收

- 12 位大师均可一对一对话。
- 圆桌、@ 指定、摘要、历史、删除、Word 导出无回归。
- 一对一流式可取消、可恢复、失败不重复。
- 分享卡片、章节和 JSON 导入导出按阶段目标通过。
- 深色模式跟随心镜全局主题。

### 8.3 数据与隐私验收

- 网页历史导入必须显式触发并可预览。
- API Key、账务数据、来访者身份不进入分享卡片。
- 临床资料默认不注入大师对话。
- 对咨询记录和督导记录的写入必须确认。
- 账务隔离逻辑保持不变。

### 8.4 安全验收

- 远程 JavaScript：0。
- 大师业务页直接模型 fetch：0。
- 大师模块 localStorage API Key：0。
- 未校验外链入口：0。
- 未转义模型内容写入 HTML：0。

### 8.5 质量门槛

- `node scripts/self-test.js` 全绿。
- 修改的 JavaScript 全部通过 `node --check`。
- 所有 `app/*.html` 保留有效 `<body>` 与 `<script>`。
- `billing-shell.html` 的 `billableSessionsFor` / `billableSessions` 仍存在且行为不变。
- 流式、迁移、圆桌和授权至少各有一个失败路径测试。
- `git diff --check` 无格式错误。

## 九、最终建议

项目可以启动，但应按以下顺序：

1. 先统一心镜内部大师对话核心。
2. 再建立内容同步机制并补科胡特。
3. 然后扩展统一 AI 层的流式协议。
4. 最后增加分享、章节、i18n 和临床关联。

最重要的技术原则不是“尽可能复制网页端”，而是：

**保留心镜的数据、授权、安全和桌面交互边界，只把 winnicott-chat 已验证的内容质量与对话体验收敛进一个可维护的内核。**
